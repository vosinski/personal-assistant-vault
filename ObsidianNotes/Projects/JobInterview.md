# Interviewer Tricks to Reveal Real Experience

## 1. The “Explain the System Backwards” Trick

Ask the candidate to explain a system **starting from the user action and going through the entire backend flow**.

Question:

- Imagine a user clicks a button in the application. Can you walk me through what happens step by step until the response is returned?
    

Listen for:

- controller
    
- service layer
    
- model/database interaction
    
- background jobs
    
- caching
    
- external services
    
- response rendering
    

Good candidates will describe something like:

> User action → controller → service/business logic → database queries → background job or API → response.

Follow-ups:

- Where could performance problems happen in this flow?
    
- What part of this flow did you personally implement?
    

Why this works:  
People who only worked on **small tasks** usually cannot explain the **whole system flow**.

---

# 2. The “Show Me the Hardest Problem” Trick

Ask the candidate about **the single hardest problem they solved**.

Question:

- What was the most technically difficult problem you personally solved in your last project?
    

Follow-ups:

- How did you discover the root cause?
    
- What solutions did you consider?
    
- Why did you choose the final solution?
    
- How long did it take to solve?
    

What you want to hear:

- investigation steps
    
- debugging process
    
- trade-offs
    
- collaboration with teammates
    
- measurable impact
    

Red flag answers sound like:

> "It was a bug and I fixed it."

Strong candidates describe **the process of solving the problem**, not just the result.

---

# 3. The “Change the Requirements” Trick

After the candidate explains a feature they built, **change the requirements slightly**.

Example:

- You mentioned building an audit logging feature.  
    What if the system suddenly needed to support **10 times more users**?
    

Follow-ups:

- What parts of the system might break first?
    
- How would you redesign the solution?
    
- Would you change the database structure or architecture?
    

Why this works:  
This reveals whether the candidate:

- understands **system scalability**
    
- understands **why their solution worked**
    
- can **adapt architecture to new constraints**
    

Junior or weak candidates usually answer:

> "I would optimize the code."

Strong candidates talk about:

- database indexes
    
- async processing
    
- caching
    
- queues
    
- architecture changes.
    

---

# Bonus Quick Test (Very Powerful)

Ask this simple question:

> When you join a new Rails project, what are the first three things you check in the codebase?

Experienced developers usually answer something like:

- Gemfile
    
- routes.rb
    
- application controller / base services
    
- database schema
    
- background jobs
    
- architecture patterns
    

Inexperienced developers struggle to answer.
